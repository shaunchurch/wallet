  <action>
Create `src/features/wallet/crypto/vault.ts` with the following:

**Imports:** `bytesToHex`, `hexToBytes` from `@noble/hashes/utils`. Types from `../types.ts`.

**Constants:**
- `PBKDF2_ITERATIONS = 600_000`
- `SALT_BYTES = 16`
- `IV_BYTES = 12`

**`deriveKey(password: string, salt: Uint8Array): Promise<CryptoKey>`**
1. Normalize password with `password.normalize('NFKD')` before encoding (handles unicode edge cases per Pitfall 4 in research).
2. `crypto.subtle.importKey('raw', encodedPassword, 'PBKDF2', false, ['deriveKey'])`
3. `crypto.subtle.deriveKey({ name: 'PBKDF2', salt, iterations: PBKDF2_ITERATIONS, hash: 'SHA-256' }, keyMaterial, { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt'])`

**`encryptVault(plaintext: VaultPlaintext, password: string): Promise<VaultBlob>`**
1. Generate random salt: `crypto.getRandomValues(new Uint8Array(SALT_BYTES))`
2. Generate random IV: `crypto.getRandomValues(new Uint8Array(IV_BYTES))`
3. Derive key from password + salt
4. Encode plaintext as UTF-8: `new TextEncoder().encode(JSON.stringify(plaintext))`
5. `crypto.subtle.encrypt({ name: 'AES-GCM', iv }, key, encoded)`
6. Return `VaultBlob` with version 1, KDF params, cipher params, hex-encoded ciphertext

**`decryptVault(blob: VaultBlob, password: string): Promise<VaultPlaintext>`**
1. Validate `blob.version === 1` (throw if unknown version)
2. Decode salt and IV from hex
3. Derive key from password + salt (using blob.kdf.iterations, NOT the constant -- enables migration)
4. `crypto.subtle.decrypt({ name: 'AES-GCM', iv }, key, ciphertext)`
5. Parse JSON from decrypted UTF-8 bytes
6. If decrypt fails (wrong password), `crypto.subtle.decrypt` throws `OperationError`. Catch it and throw `new Error('Incorrect password')` -- never expose internal error details.

**`createLockoutManager(state?: LockoutState): LockoutManager`** -- factory function returning lockout state + methods. Pass saved state to restore from persistence:
```typescript
interface LockoutManager {
  checkLockout(): { locked: boolean; remainingMs: number };
  recordFailure(): void;
  reset(): void;
  serialize(): LockoutState;
}
// Restore via factory: createLockoutManager(savedState)
```
- Track `failedAttempts` and `lockedUntil` timestamp
- After 3 failures: 5s delay. After 4: 15s. After 5+: 30s.
- `checkLockout()` returns whether currently locked and remaining ms
- `reset()` clears on successful unlock
- `serialize()`/`restore()` for chrome.storage.session persistence (survives SW termination per Pitfall 1)

**Anti-patterns enforced:**
- NEVER reuse IV -- fresh `crypto.getRandomValues` every encrypt call
- NEVER log password or plaintext bytes
- ALWAYS use blob.kdf.iterations for decrypt (not hardcoded constant)
- ALWAYS normalize password NFKD

**Test suite: "vault wrong password"**
- Encrypt with "correct", decrypt with "wrong" -> throws error matching "Incorrect password"
- Encrypt with "correct", decrypt with "" -> throws
- Encrypt with "", decrypt with "x" -> throws

**Test suite: "vault blob structure"**
- Encrypt and inspect returned VaultBlob:
  - `version` is 1
  - `kdf.algorithm` is "PBKDF2"
  - `kdf.hash` is "SHA-256"
  - `kdf.iterations` >= 600_000
  - `kdf.salt` is 32-char hex string (16 bytes)
  - `cipher.algorithm` is "AES-256-GCM"
  - `cipher.iv` is 24-char hex string (12 bytes)
  - `data` is hex string (non-empty)

**Test suite: "vault unique salt and IV"**
- Encrypt same plaintext with same password twice
- Assert salt differs between the two blobs
- Assert IV differs between the two blobs
- Both decrypt correctly

**Test suite: "lockout manager"**
- Fresh manager: not locked
- 1-2 failures: not locked
- 3rd failure: locked for 5s (`checkLockout().locked === true`, `remainingMs > 0 && remainingMs <= 5000`)
- `reset()`: clears lockout
- `serialize()` -> `restore()` round-trips state

Run `pnpm test -- tests/crypto/vault.test.ts`. All must pass.

**Performance note:** 600k PBKDF2 iterations take ~200-500ms per operation in Node.js with native crypto. Tests with 10 password variants = ~5-10s total. This is expected. Do NOT reduce iterations for tests -- we're testing the real implementation.
